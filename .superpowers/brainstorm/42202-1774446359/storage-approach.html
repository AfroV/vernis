Where should saved generators live?

Each approach has different trade-offs for integration, complexity, and separation

A

Real HTML File in NFT Directory

Create a small .html file (e.g. gazer-247.html) in /opt/vernis/nfts/ that loads the generator

Pros

  • Zero special-casing — works with Manage Art, carousels, hide/show, delete out of the box
  • Gallery already handles iframes for .html files
  • Shows up in file count, storage stats, backup/restore
  • One source of truth — no second list to keep in sync

Cons

  • nft-list-detailed endpoint currently skips .html — needs extending
  • Thumbnail in Manage Art grid will be blank/generic (no image preview)
  • Mixes generated content with downloaded IPFS art on disk
  • Filename collisions possible if not careful with naming
B

Separate Generators JSON File

New file /opt/vernis/gallery-generators.json with saved entries. Gallery merges both lists.

Pros

  • Clean separation — IPFS art stays pure, generators are their own thing
  • Easy to store rich metadata (token number, type, source URL, preview image URL)
  • Can show Art Blocks preview thumbnails in Manage Art (static image URL is known)
  • No disk clutter — one JSON file vs many small .html files

Cons

  • Gallery loader needs to merge two sources — more complex
  • Manage Art needs special handling for generator entries (different delete, no CID, etc.)
  • Carousels, hide/show, backup all need to know about generators separately
  • Second source of truth — potential sync issues
C

Entries in Display Config

Add a generators array to display-config.json

Pros

  • Minimal new infrastructure — no new files or endpoints
  • Gallery already loads display-config every 30s
  • Simple to implement quickly

Cons

  • Mixes content with display preferences — bad separation of concerns
  • No Manage Art integration at all — can't hide, carousel, or delete from UI
  • display-config POST overwrites the whole file — risk of data loss
  • Doesn't scale — becomes unwieldy as more generators are added
